# AGENTS.md — EastMesh ESP32 Flasher

Guide for AI agents and contributors working in this repository.

## Project Overview

Browser-based ESP32 firmware flasher and serial monitor for the
[EastMesh](https://github.com/xJARiD/MeshCore-EastMesh) project. The app lets
users select firmware from a local manifest, flash it to an ESP32 board over
Web Serial, and interact with the device through a two-way serial console.

## Repository Layout

```
├── index.html                 # Single-page HTML shell (dark-themed, all CSS inline)
├── app.js                     # Application logic (ES6 module, vanilla JS)
├── sync_meshcore_releases.py  # CLI tool: syncs firmware binaries + builds manifest
├── firmwares/
│   └── manifest.json          # Auto-generated firmware catalog (DO NOT hand-edit)
├── THEME_STYLE_GUIDE.md       # EastMesh brand tokens and visual language
└── README.md                  # User-facing documentation
```

Firmware `.bin` files live under `firmwares/<board>/<firmware>/<version>/` but
are git-ignored (`.gitignore` excludes `firmwares/**/*.bin`).

## Tech Stack

| Layer       | Technology                                          |
|-------------|-----------------------------------------------------|
| Frontend    | Vanilla HTML / CSS / JavaScript (no build step)     |
| Flash API   | [esptool-js v0.6.0](https://unpkg.com/esptool-js@0.6.0/bundle.js) via CDN |
| Serial API  | Web Serial API (Chromium-based browsers only)       |
| Sync script | Python 3 (stdlib only — no pip dependencies)        |
| Hosting     | Any static HTTP server (`python3 -m http.server`)   |

## Architecture

### Dropdown Cascade (app.js)

The firmware selection UI follows this dependency chain:

```
Firmware → Board → Version → Image Type
```

Each dropdown change triggers a refresh of its dependents:

- `firmwareSelect` change → `refreshBoards()` → `refreshVersions()` → `refreshImageTypes()`
- `boardSelect` change → `refreshVersions()` → `refreshImageTypes()`
- `versionSelect` change → `refreshImageTypes()`

The helper `setOptions(select, options, preferredValue)` rebuilds a `<select>`
element's options and preserves the previous selection when possible.

### Flasher Lifecycle

The flasher auto-connects on first use (Flash or Erase action) via
`connectFlasherIfNeeded()` and auto-disconnects after the operation completes
via `safelyDisconnectFlasher()`. There is no persistent "Connect" button.

### Serial Console

The serial console is independent from the flasher — it uses a separate port
(`serialPort`) and can run concurrently. Connection is manual via Connect /
Disconnect buttons.

### Firmware Manifest (`firmwares/manifest.json`)

Generated by `sync_meshcore_releases.py`. Structure:

```
boards → {board_key} → firmwares → {firmware_key} → versions → {version_key} → images → {image_type}
```

Image types: `app_bin` (address `0x10000`) and `merged_full_flash` (address `0x0000`).

## Coding Conventions

### JavaScript (app.js)

- **No framework, no build step.** Keep it vanilla ES6 modules.
- **Import from CDN only.** Do not add npm / package.json.
- **DOM references** are declared as top-level `const` bindings from `getElementById`.
- **Event listeners** are registered at module scope after function definitions.
- **Async error handling:** wrap `await` calls in try/catch; surface errors to
  the log area (`appendLog`) or console area (`appendConsole`).
- **Naming:** camelCase for variables/functions. Prefix booleans with
  `is`/`has`/`keep` (e.g., `serialKeepReading`).

### HTML/CSS (index.html)

- **All styles are inline** in `<style>` within `<head>`.
- **CSS custom properties** (design tokens) are defined on `:root`.
- **Dark-first design.** Follow `THEME_STYLE_GUIDE.md` for colors and tokens.
- **No external CSS frameworks** in the current build. BeerCSS references in
  the style guide are for other EastMesh surfaces.

### Python (sync_meshcore_releases.py)

- **Standard library only.** Do not add third-party dependencies.
- **Regex-based filename parsing** via `FILENAME_RE`. Update the regex when
  upstream naming conventions change.
- **Idempotent syncing:** existing files are skipped; manifest is rebuilt from
  whatever files are present locally.

## Common Tasks

### Sync firmware and rebuild manifest

```bash
python3 sync_meshcore_releases.py
```

### Rebuild manifest without downloading

```bash
python3 sync_meshcore_releases.py --manifest-only
```

### Run locally

```bash
python3 -m http.server 8000
# Open http://localhost:8000 in Chrome or Edge
```

### Add a new firmware type

1. Update `FILENAME_RE` in `sync_meshcore_releases.py` to match the new naming pattern.
2. Re-run the sync script.
3. Verify the new firmware appears in the UI dropdowns.

### Update theme colors

1. Update `THEME_STYLE_GUIDE.md` first.
2. Update CSS custom properties in `index.html` `:root`.
3. Verify focus, hover, and active states across all controls.

## Testing

There is no automated test suite. Verify changes manually:

1. Load the page and confirm dropdowns populate from manifest.
2. Flash a board and confirm progress + completion log.
3. Erase a board and confirm completion log.
4. Connect serial, send a command, and verify console output.
5. Download firmware and confirm the correct `.bin` file is saved.

## Important Notes

- **Do not hand-edit `firmwares/manifest.json`.** It is generated by the sync script.
- **Do not commit `.bin` files.** They are git-ignored.
- **Web Serial requires HTTPS or localhost.** The app won't work from `file://`.
- **Only Chromium-based browsers** support Web Serial (Chrome, Edge, Opera).
